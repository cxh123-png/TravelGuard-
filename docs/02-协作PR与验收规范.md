# TravelGuard 协作、PR 与验收规范

## 1. 分支规则

- `main` 只包含已审阅、已验证、可集成的内容；任何成员和 AI 都不得直接推送。
- 每项工作先有 GitHub Issue，再从最新 `main` 创建一个短生命周期分支。
- 分支名使用 `feat/<issue>-<module>`、`fix/<issue>-<module>`、`docs/<issue>-<topic>` 或 `test/<issue>-<topic>`。
- 每个 PR 只对应一个 Issue；跨模块变更必须拆成“契约 PR”与“实现 PR”。
- 合并使用 squash merge，合并后删除远程功能分支。

## 2. 成员与 AI 的工作流程

```text
git switch main
git pull --ff-only origin main
git switch -c feat/<issue>-<module>
```

将以下材料一起交给 AI：根目录 `实现方案.md`、本文档、自己的任务包、对应 Issue，以及当前分支的实际代码。明确告诉 AI：不得修改任务包未授权的目录；若缺少前置接口，停止实现并在 PR/Issue 中说明阻塞条件。

AI 可以实现、测试和解释本模块，但不能代替成员确认业务结果，也不能直接推送或合并 `main`。

## 3. PR 必填内容

PR 描述必须包含：

1. 关联 Issue 编号和任务包编号。
2. 本次实现的用户可观察行为。
3. 修改文件列表，以及是否触及公共 Schema。
4. 已运行命令及真实输出摘要。
5. 至少一个验收证据：API 请求/响应、页面截图、Agent Trace 或测试报告。
6. 已知限制或被前置依赖阻塞的内容。

## 4. 合并判定

| 检查项 | 通过条件 |
|---|---|
| 范围 | 只修改任务包授权目录；公共契约变化已获相关模块同意 |
| 依赖 | 所有“开始条件”已在 `main` 合并；未用本地临时代码替代依赖 |
| 自动检查 | Ruff、mypy、受影响的 pytest、前端 lint/typecheck/test 均通过 |
| 业务验收 | 任务包指定的 HTTP、页面或 Agent 场景有真实证据 |
| 安全 | 无密钥、真实个人数据、任意执行工具或绕过状态机的实现 |
| 审阅 | 至少一位非作者审阅通过；所有评论已解决 |

只有六项全部满足时，维护者才合并。维护者不需要逐行判断每段代码：以任务包验收标准、自动检查、可观察演示和审阅意见作为决策依据；需要代码层复核时，可将 PR 交给 Codex 按任务包审查。

## 5. 冲突与依赖处理

- 需要别人的接口：先在 Issue 写明所需输入、输出、错误码和完成时间；不自行实现对方模块。
- 两个模块都需要改公共 Schema：先创建一个只修改 Schema 与契约测试的 PR，合并后各自更新实现。
- 依赖未合并但需要提前工作：只提交 Mock、测试数据、页面骨架或契约测试；PR 标记 Draft。
- 合并冲突：先同步 `main`，由冲突文件的模块 owner 决定业务含义；不要为了消除冲突删掉测试或逻辑。

## 6. Issue 模板

```markdown
## 目标

## 所属模块与任务包

## 开始条件

## 允许修改的目录

## 不允许修改的目录

## 交付物

## 验收场景与命令

## 解锁的后续任务
```
